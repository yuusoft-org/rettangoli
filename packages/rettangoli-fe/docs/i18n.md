# Internationalization Spec (`i18n`)

This document defines the normative internationalization contract for
`rettangoli-fe`.

## 1. Scope

Internationalization is a frontend framework feature. It covers:
1. project-level i18n configuration
2. one locale source file per language
3. build-time generation of lazy-loadable JSON locale assets
4. runtime injection of `i18n` and `locale`
5. automatic re-rendering when the active locale changes

Application code MUST NOT pass `i18n` or `locale` manually through `setup.js`.
The FE runtime owns these dependencies in the same way it owns `store` and
`render`.

## 2. Configuration

i18n configuration belongs under `fe.i18n` in `rettangoli.config.yaml`.

```yaml
fe:
  dirs:
    - "src/components"
    - "src/pages"
  setup: "src/setup.web.js"
  outfile: "_site/public/main.js"
  i18n:
    dir: "src/i18n"
    defaultLocale: "en"
    fallbackLocale: "en"
    locales:
      - "en"
      - "vi"
      - "ja"
```

| Key | Required | Description |
| --- | --- | --- |
| `dir` | Yes | Directory containing locale YAML files |
| `defaultLocale` | Yes | Initial locale used by the runtime |
| `fallbackLocale` | Yes | Locale used when another locale cannot be loaded |
| `locales` | Yes | Ordered list of supported locale codes |

Locale names MUST match the source filenames in `dir`.

## 3. Locale Files

Projects MUST use one YAML file per locale.

```text
src/i18n/
  en.yaml
  vi.yaml
  ja.yaml
```

The locale is the filename. The file content MUST use exactly two key levels:

```yaml
namespace:
  key: "content"
```

Example:

```yaml
common:
  saveButton: "Save"
  cancelButton: "Cancel"

projectsPage:
  title: "Projects"
  emptyState: "No projects yet"

commandLineConditional:
  branchesLabel: "Branches"
```

Valid namespaces:
- `common`: generic UI text
- `domain`: shared product vocabulary
- component namespaces such as `commandLineConditional`
- page namespaces such as `projectsPage`
- `rtglUi`: optional app overrides for Rettangoli UI copy

The hierarchy MUST NOT exceed two levels.

Valid:

```yaml
common:
  saveButton: "Save"
```

Invalid single-level key:

```yaml
saveButton: "Save"
```

Invalid three-level key:

```yaml
common:
  actions:
    save: "Save"
```

Leaf values MUST be strings.

## 4. View Usage

Views read translated strings through the framework-provided `i18n` object.
`i18n` is supported in template content, template bindings, and event listener
payloads.

```yaml
template:
  - rtgl-text s=h1: ${i18n.projectsPage.title}
  - rtgl-button: ${i18n.common.saveButton}
```

View access MUST always use two levels after `i18n`.

Valid:

```yaml
- rtgl-text s=h1: ${i18n.commandLineConditional.branchesLabel}
```

Invalid:

```yaml
- rtgl-text s=h1: ${i18n.branchesLabel}
- rtgl-text s=h1: ${i18n.common.actions.save}
```

## 5. Dynamic Text

Parameterized message functions are not part of the i18n template contract.

Invalid:

```yaml
- rtgl-button: ${i18n.completedCount({ count: completedCount })}
```

Invalid:

```yaml
- rtgl-button: ${__rtglI18n("completedCount", completedCount)}
```

Apps that need dynamic text SHOULD compute that string in JavaScript before it
reaches the view.

```js
export const selectViewData = ({ state, i18n }) => ({
  completedLabel: `${state.completedCount} ${i18n.todos.completedSuffix}`,
});
```

```yaml
template:
  - rtgl-button: ${completedLabel}
```

Event listener payloads can also read translated strings:

```yaml
refs:
  saveButton:
    eventListeners:
      click:
        handler: handleSave
        payload:
          label: ${i18n.common.saveButton}
```

## 6. Runtime Dependencies

When i18n is enabled, FE MUST inject:

| Dependency | Available in | Description |
| --- | --- | --- |
| `i18n` | views, handlers, store context | Current locale messages |
| `locale` | handlers, store context | Locale service |

Store context SHOULD include `i18n` and `locale`:

```js
export const selectViewData = ({ i18n }) => ({
  title: i18n.projectsPage.title,
});
```

Handlers SHOULD receive `i18n` and `locale` through `deps`:

```js
export const handleSubmit = ({ i18n }) => {
  console.log(i18n.common.saveButton);
};
```

## 7. Locale Service

The locale service MUST expose:

```js
locale.current();      // current locale string
locale.available();    // supported locale strings
await locale.load("vi");
await locale.set("vi");
```

`locale.set(nextLocale)` MUST:
- validate that `nextLocale` is configured
- lazy-load that locale's JSON asset if needed
- cache the loaded catalog
- update the active locale
- re-render mounted FE components

The runtime starts from `defaultLocale`. It does not persist locale selection.
If an app wants to remember or derive a user's preferred locale, that state is
app-owned and should eventually call `locale.set(nextLocale)`.
Page-level metadata such as `<html lang>` is also app-owned.

## 8. Changing Locale

Use a normal event handler. The app does not import an i18n library and does
not pass locale state through props.

```yaml
refs:
  languageSelect:
    eventListeners:
      change:
        handler: handleChangeLocale
        payload:
          nextLocale: ${_event.target.value}

template:
  - select#languageSelect:
      - option value=en: "English"
      - option value=vi: "Vietnamese"
      - option value=ja: "Japanese"
```

```js
export const handleChangeLocale = async ({ locale }, { nextLocale }) => {
  await locale.set(nextLocale);
};
```

The runtime MUST re-render all mounted components after `locale.set()` resolves.

## 9. Build Output

The FE build MUST read locale YAML files and generate runtime JSON assets next
to the configured FE bundle.

For this config:

```yaml
fe:
  outfile: "_site/public/main.js"
  i18n:
    dir: "src/i18n"
    locales: ["en", "vi"]
```

The build output MUST include:

```text
_site/public/main.js
_site/public/i18n/en.json
_site/public/i18n/vi.json
```

The runtime SHOULD resolve locale JSON URLs relative to the generated FE bundle
so apps do not need to configure a separate public path.

## 10. Validation

The build/check pipeline MUST reject:
- missing configured locale files
- locale files whose root is not an object
- single-level i18n keys
- namespace values that are not objects
- nested values deeper than `namespace -> key -> content`
- non-string leaf values
- view expressions that use `${i18n.someKey}`
- view expressions that use `${i18n.a.b.c}`
- view expressions that call i18n functions

The build/check pipeline SHOULD report missing keys between locales using
`defaultLocale` as the source key set.
