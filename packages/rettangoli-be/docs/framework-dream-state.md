# rettangoli-be Dream State

Status: draft

This document describes the best possible end-state for `rettangoli-be`.

It is not limited by the current implementation. It is the target we design toward.

Dream-state names use the contract package convention: `.contract.yaml` for
contracts, `.examples.yaml` for executable examples, and `.rtgl/` for generated
artifacts.

## North Star

`rettangoli-be` is an end-to-end, contract-first backend framework for long-lived
JSON-RPC APIs.

Humans and AI agents build APIs by editing contracts, executable examples, and small
method implementations. The framework owns the full first-party path:

```txt
idea
  -> contract package
  -> executable examples
  -> implementation
  -> manifest
  -> compatibility report
  -> generated docs
  -> verification proof
  -> running API
```

Contracts are the product. Implementation is the proof.

## 20-Pass Design Loop

This dream state was reviewed through these lenses:

1. constitution alignment;
2. one-path simplicity;
3. end-to-end framework ownership;
4. framework policy written by us;
5. library use only at hard boundaries;
6. standards compatibility;
7. contract primacy;
8. executable examples as contract tests;
9. public error catalog;
10. canonical manifest;
11. compatibility over years;
12. generated docs artifacts;
13. closed-loop agent verification;
14. long-running agent loops without drift;
15. parallel agent work units;
16. evidence and proof artifacts;
17. SQLite migration lifecycle;
18. explicit transaction and DAO boundaries;
19. build/runtime separation;
20. release and CI gates.

## Best State In One Sentence

An agent can take a bounded API change, claim the work, update the contract package,
run deterministic checks, repair failures, generate artifacts, prove compatibility,
and produce a machine-readable verification bundle that another agent or human can
trust.

## Source And Generated Artifacts

The framework should make source and generated files explicit.

| Path | Kind | Owner | Editable | Generated By |
| --- | --- | --- | --- | --- |
| `src/modules/<domain>/<action>/<action>.contract.yaml` | source | app | yes | scaffold |
| `src/modules/<domain>/<action>/<action>.examples.yaml` | source | app | yes | scaffold |
| `src/modules/<domain>/<action>/<action>.handlers.js` | source | app | yes | scaffold |
| `src/middleware/<name>.js` | source | app | yes | scaffold |
| `src/setup.js` | source | app | yes | scaffold |
| `migrations/*.sql` | source | app | yes | migration scaffold |
| `.rtgl/generated/registry.js` | generated | framework | no | `rtgl be build` |
| `.rtgl/generated/app.js` | generated | framework | no | `rtgl be build` |
| `.rtgl/manifest.json` | generated | framework | no | `rtgl be manifest` |
| `.rtgl/compat/*.json` | generated | framework | no | `rtgl be compat` |
| `.rtgl/evidence/<taskId>/*` | generated | framework | no | `rtgl be verify` |
| generated API docs | generated | framework | no | `rtgl be docs` |

Generated files are deterministic. Humans and agents review generated diffs, but do
not hand-edit them.

## Method Contract Package

Every API method is one isolated package:

```txt
src/modules/user/getProfile/
  getProfile.contract.yaml
  getProfile.examples.yaml
  getProfile.handlers.js
```

The package contains everything needed to understand, review, test, implement, and
evolve one method.

Review order:

1. contract diff;
2. executable example coverage;
3. manifest diff;
4. compatibility report;
5. generated docs diff;
6. handler implementation.

Implementation is last.

## Contract File

The contract file is the public API definition.

It should be versioned, lintable, comparable, and generator-safe.

```yaml
schemaVersion: rettangoli.contract/v1
method: user.getProfile
summary: Get profile
description: Return the authenticated user profile.

middleware:
  before: [withAuthUser]
  after: [withLogger]

params:
  schema:
    type: object
    additionalProperties: false
    properties: {}
    required: []

result:
  schema:
    type: object
    additionalProperties: false
    properties:
      id:
        type: string
      email:
        type: string
      role:
        type: string
    required: [id, email, role]

errors:
  AUTH_REQUIRED:
    category: auth
    description: Session is missing or invalid.
    publicMessage: Authentication is required.
    retryable: false
    visibility: public
    jsonRpc:
      code: -32000
    details:
      schema:
        type: object
        additionalProperties: false
        properties:
          reason:
            const: auth_required
        required: [reason]
  USER_NOT_FOUND:
    category: domain
    description: Authenticated user no longer exists.
    publicMessage: User was not found.
    retryable: false
    visibility: public
    jsonRpc:
      code: -32000
    details:
      schema:
        type: object
        additionalProperties: false
        properties:
          userId:
            type: string
        required: [userId]
```

Rules:

- schema version is required;
- method id matches folder path;
- JSON Schema dialect is fixed by the framework and recorded in the manifest;
- expected errors are a named public catalog;
- the contract stays focused on public API shape and expected public failures;
- operational policy is not added to the core contract unless it proves necessary.

The minimal contract core is:

- `schemaVersion`;
- `method`;
- `description`;
- `middleware`;
- `params`;
- `result`;
- `errors`.

Everything else should be earned by real use, not added as default ceremony.

## Executable Examples

Examples are contract tests, not just implementation tests.

```yaml
schemaVersion: rettangoli.examples/v1
file: './getProfile.handlers.js'
group: user-get-profile
---
suite: userGetProfileMethod
exportName: userGetProfileMethod
---
case: returns-profile
proves:
  result: success
request:
  jsonrpc: '2.0'
  id: profile-ok
  method: user.getProfile
  params: {}
context:
  authUser:
    userId: u-1
out:
  jsonrpc: '2.0'
  id: profile-ok
  result:
    id: u-1
    email: demo@example.com
    role: user
---
case: requires-auth
proves:
  error: AUTH_REQUIRED
request:
  jsonrpc: '2.0'
  id: profile-auth
  method: user.getProfile
  params: {}
out:
  jsonrpc: '2.0'
  id: profile-auth
  error:
    code: -32000
    message: Domain error
    data:
      code: AUTH_REQUIRED
      details:
        reason: auth_required
```

The framework proves:

- request params match the params schema;
- success response results match the result schema;
- error response data matches the named error catalog;
- error details match the error details schema;
- every method has at least one explicit success proof;
- every public error has at least one example;
- examples run through the JSON-RPC app runtime;
- examples can run repeatedly with deterministic results.

## Handler File

Handlers stay small and boring.

```js
export const userGetProfileMethod = async ({ context, deps }) => {
  if (!context.authUser?.userId) {
    return {
      _error: true,
      code: 'AUTH_REQUIRED',
      details: { reason: 'auth_required' },
    };
  }

  const user = await deps.userDao.findById({ userId: context.authUser.userId });

  if (!user) {
    return {
      _error: true,
      code: 'USER_NOT_FOUND',
      details: { userId: context.authUser.userId },
    };
  }

  return {
    id: user.id,
    email: user.email,
    role: user.role,
  };
};
```

Handlers do not know HTTP, JSON-RPC envelopes, generated docs, environment
variables, or database connection construction.

## Manifest

The manifest is the canonical machine-readable API artifact.

It is deterministic JSON:

- sorted keys;
- stable schema version;
- no timestamps unless explicitly provided by build metadata;
- source paths included;
- content hashes included;
- full client-facing schemas included.

```json
{
  "schemaVersion": "rettangoli.manifest/v1",
  "framework": "@rettangoli/be",
  "frameworkVersion": "x.y.z",
  "jsonSchemaDialect": "https://json-schema.org/draft/2020-12/schema",
  "app": {
    "name": "routevn-api",
    "version": "0.1.0"
  },
  "methods": {
    "user.getProfile": {
      "source": {
        "contract": "src/modules/user/getProfile/getProfile.contract.yaml",
        "examples": "src/modules/user/getProfile/getProfile.examples.yaml",
        "handler": "src/modules/user/getProfile/getProfile.handlers.js"
      },
      "hashes": {
        "contract": "sha256:...",
        "examples": "sha256:...",
        "handler": "sha256:..."
      },
      "params": {},
      "result": {},
      "errors": {
        "AUTH_REQUIRED": {
          "category": "auth",
          "retryable": false,
          "visibility": "public",
          "detailsSchema": {}
        },
        "USER_NOT_FOUND": {
          "category": "domain",
          "retryable": false,
          "visibility": "public",
          "detailsSchema": {}
        }
      },
      "examples": {
        "success": 1,
        "errors": {
          "AUTH_REQUIRED": 1,
          "USER_NOT_FOUND": 1
        }
      }
    }
  }
}
```

Agents should understand most of the API from the manifest without loading every
source file into context.

## Compatibility

Compatibility checks compare manifests.

Breaking changes:

- remove a method;
- rename a method;
- add required params;
- narrow params;
- remove result fields;
- change result field types;
- remove error codes;
- change error details incompatibly.

Risky changes:

- add optional params;
- add result fields;
- add error codes;
- change middleware;
- loosen `additionalProperties`;
- change nullable behavior.

Safe changes:

- improve descriptions;
- add examples;
- widen accepted params deliberately.

Compatibility report shape:

```json
{
  "schemaVersion": "rettangoli.compat/v1",
  "ok": false,
  "summary": {
    "breaking": 1,
    "risky": 0,
    "safe": 2
  },
  "findings": [
    {
      "ruleId": "RTGL-COMPAT-ERROR-REMOVED",
      "severity": "breaking",
      "method": "user.getProfile",
      "jsonPointer": "/methods/user.getProfile/errors/USER_NOT_FOUND",
      "before": { "exists": true },
      "after": { "exists": false },
      "rationale": "Removing a public error code can break clients.",
      "requiresOverride": true
    }
  ]
}
```

Breaking changes require explicit review metadata.

## Generated Docs

Generated docs come from the manifest and examples.

Docs include:

- method summary and description;
- params schema;
- result schema;
- error catalog;
- examples.

Generated docs are stale if their recorded manifest hash does not match the current
manifest.

## Agent Task State

Agents need first-class task state for long-running loops.

```txt
.rtgl/tasks/<taskId>.json
.rtgl/evidence/<taskId>/
  verify.json
  manifest.json
  compat.json
  checks/
    check.json
    test.json
  summary.md
```

Task state:

```json
{
  "schemaVersion": "rettangoli.task/v1",
  "taskId": "task-user-get-profile",
  "goal": "Add user.getProfile",
  "scope": {
    "methods": ["user.getProfile"],
    "migrations": [],
    "generatedTargets": ["manifest", "docs"]
  },
  "status": "in_progress",
  "plan": [
    { "id": "contract", "status": "done" },
    { "id": "examples", "status": "done" },
    { "id": "handler", "status": "in_progress" },
    { "id": "verify", "status": "pending" }
  ],
  "anchors": {
    "baseManifestHash": "sha256:...",
    "currentManifestHash": "sha256:...",
    "contractHashes": {
      "user.getProfile": "sha256:..."
    }
  },
  "nextAction": "run rtgl be verify --method user.getProfile --json"
}
```

Task state lets agents compact context, resume later, and stay anchored to the same
goal.

## Drift Resistance

Long-running agent loops must resist drift.

The framework should track:

- base manifest hash;
- current manifest hash;
- contract hashes;
- example hashes;
- handler hashes;
- migration checksums;
- generated artifact hashes;
- task state hash.

Resume preflight:

```bash
rtgl be resume --task task-user-get-profile --json
```

Preflight reports:

- files changed since last checkpoint;
- generated artifacts now stale;
- manifest drift;
- migration drift;
- compatibility status changed;
- task scope conflicts.

Agents should never have to guess whether their context is stale.

## Parallel Agents

Parallel work is safe when ownership is explicit.

Safe work units:

- one method package;
- one middleware;
- one migration;
- one compatibility report;
- one domain docs section.

Ownership claim:

```json
{
  "schemaVersion": "rettangoli.claim/v1",
  "taskId": "task-user-get-profile",
  "owns": {
    "methods": ["user.getProfile"],
    "migrations": [],
    "generatedTargets": ["manifest", "docs"]
  }
}
```

Merge gates:

- no duplicate method ids;
- no overlapping ownership claims;
- no stale generated manifests;
- no stale generated docs;
- no conflicting migrations;
- no unreviewed breaking changes;
- all examples prove declared contracts.

Parallel agents coordinate through claims, manifests, compatibility reports, and
verification proofs.

## Checkpoints And Recovery

Risky edits should have checkpoints.

Ideal commands:

```bash
rtgl be checkpoint --task task-user-get-profile --json
rtgl be diff --checkpoint <checkpointId> --json
rtgl be restore-plan --checkpoint <checkpointId> --json
```

The framework should not hide destructive operations, but it should give agents enough
metadata to recover cleanly.

## SQLite Dream State

SQLite is first-class, explicit, and visible.

It is not hidden behind an ORM.

### Migration Lifecycle

Migrations are plain SQL files:

```txt
migrations/
  0001_create_users.sql
  0002_create_projects.sql
```

Rules:

- migration files are ordered;
- migration ids are stable;
- checksums are recorded;
- applied migrations are recorded in a ledger table;
- migrations are replayable from an empty database;
- destructive migrations require explicit metadata;
- framework helpers can scaffold/check migrations but do not become a schema builder.

Migration metadata:

```yaml
schemaVersion: rettangoli.migration/v1
id: 0001_create_users
file: migrations/0001_create_users.sql
checksum: sha256:...
destructive: false
tables:
  creates: [users]
  alters: []
  drops: []
```

### Database Review

Database behavior should stay visible, but not as required per-method contract
ceremony.

The primary review artifacts are:

- SQL migrations;
- DAO code;
- executable examples;
- integration flows when a method changes data.

The framework checks:

- migrations are ordered and replayable;
- destructive migrations are explicit;
- tests use isolated databases;
- generated manifests are current;
- agents can inspect migrations and DAO code before implementation review.

### Transactions

Transactions are explicit.

Rules:

- write methods declare whether a transaction is required;
- DAOs accept `db` or `tx` explicitly;
- nested write workflows use savepoints when supported;
- retry behavior for `SQLITE_BUSY` and remote conflicts is declared by the app;
- batch JSON-RPC does not exist unless the framework explicitly supports it.

### Test Databases

Tests use isolated databases.

Rules:

- no test uses the shared dev database;
- test DBs are temporary;
- migrations replay from zero;
- fixtures are deterministic;
- `PRAGMA foreign_keys = ON` is enabled;
- cleanup is automatic and visible in test output.

### Local, Dev, Prod

Conventions:

- local DB path is predictable;
- dev reset is explicit;
- production can target SQLite-compatible services such as libSQL/Turso;
- production config is explicit;
- migration locking is defined;
- backup expectations are documented;
- compatibility checks catch SQL that works locally but not on the target.

### No ORM Boundary

The framework does not expose:

- entity managers;
- model classes;
- query builder APIs;
- multi-database abstraction;
- generated ORM layers.

DAOs are thin SQL wrappers owned by the app.

## Setup, Dependencies, Middleware, Extensions

`src/setup.js` is the composition root.

Setup owns:

- config;
- logger;
- database clients;
- DAO construction;
- external service clients;
- lifecycle hooks;
- extension registration;
- observability hooks.

Setup does not own:

- method discovery;
- handler aggregation;
- JSON-RPC dispatch;
- generated registries.

Middleware:

- is named by file;
- is referenced by contracts;
- is domain-agnostic;
- mutates request context deliberately;
- does not hide business policy outside contract metadata.

Extensions:

- are explicit;
- have unique names and paths;
- have setup and shutdown hooks;
- cannot mutate the RPC contract model;
- share framework observability and error policy.

## Build-Time, Startup, Request-Time

The framework should separate concerns clearly.

Check/build time:

- parse contracts;
- validate schemas;
- validate examples;
- generate registry;
- generate manifest;
- generate docs;
- run compatibility checks.

Startup time:

- load setup;
- load deterministic registry/manifest;
- verify no drift;
- initialize lifecycle hooks;
- fail before serving if contract state is invalid.

Request time:

- parse JSON-RPC;
- validate params;
- run middleware;
- call handler;
- validate success/error output;
- map output to JSON-RPC;
- emit observability events.

Runtime stays boring because contracts and tooling carry the power.

## Canonical Workflows

### New Method

```bash
rtgl be new-method user getProfile --dry-run
rtgl be new-method user getProfile
rtgl be verify --method user.getProfile --json
```

Creates:

- contract file;
- examples file;
- handler stub;
- review checklist;
- manifest diff after generation.

### Change Contract

```bash
rtgl be explain user.getProfile --json
rtgl be check --method user.getProfile --json
rtgl be test --method user.getProfile --json
rtgl be manifest --json
rtgl be compat --from base.json --to .rtgl/manifest.json --json
rtgl be verify --method user.getProfile --json
```

### Add Migration

```bash
rtgl be new-migration create_users --dry-run
rtgl be new-migration create_users

```

### Release Verify

```bash
rtgl be check --json
rtgl be test --json
rtgl be manifest --json
rtgl be compat --from released.json --to .rtgl/manifest.json --json
rtgl be docs --check --json
rtgl be verify --json
```

## Release And CI Gates

A change is ready to merge when:

- all contracts are valid;
- examples prove contracts;
- implementation satisfies contracts;
- manifest is current;
- generated docs are current;
- compatibility report is safe or reviewed;
- migrations are valid;
- SQLite target compatibility passes;
- verification proof is generated.

CI should fail on stale generated artifacts, unreviewed breaking changes, missing
examples, migration drift, or non-deterministic output.

## Non-Goals

The dream state is not:

- an ORM;
- a REST framework;
- a GraphQL framework;
- a Nest/Koa clone;
- a general HTTP router;
- a universal plugin runtime;
- a multi-database abstraction;
- runtime code generation;
- a framework where every app invents its own layout.

## Best Possible State

In the best state:

- contracts are reviewed before implementation;
- executable examples define public behavior;
- error codes are cataloged and stable;
- handlers are small and boring;
- manifests are deterministic and canonical;
- compatibility is automatic;
- generated docs come from contracts;
- SQLite impact is explicit and checkable;
- agents can work for long periods without drift;
- many agents can work in parallel safely;
- every task can end with a verification proof;
- runtime stays small because the contract system carries the power.

That is the dream: an end-to-end framework where humans and agents move quickly
because the framework makes the correct path narrow, explicit, checkable, and
repeatable.
